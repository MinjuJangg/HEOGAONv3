# 허가온 V3 서비스 플로우

허가온 V3는 사용자가 자연어로 창업 또는 영업 관련 상황을 입력하면, AI가 필요한 정보를 추출하고, 주소/API/그래프/DB를 이용해 가능 여부와 준비해야 할 서류를 안내하는 서비스입니다.

예시 입력:

```text
마포구 망원동에서 15평 카페를 창업하고 싶어요.
음료와 디저트를 팔고, 간판도 달고, 가게 앞에 테이블도 두고 싶어요.
```

---

## 1. 사용자 자연어 입력

사용자는 처음부터 정해진 양식을 채우지 않아도 됩니다.

예를 들어 아래처럼 자유롭게 입력할 수 있습니다.

- "마포구 망원동에서 카페를 열고 싶어요."
- "강남구에서 일반음식점을 준비 중이고 외부 테이블도 놓고 싶어요."
- "송파구 매장에 간판만 새로 설치하고 싶어요."
- "이미 운영 중인 가게에서 주류 판매를 추가하고 싶어요."

즉, 서비스는 단순한 창업만이 아니라 아래 시나리오까지 커버하는 것을 목표로 합니다.

- 신규 창업/개업
- 업종 변경
- 주류 판매 추가
- 간판 설치
- 외부 테이블/도로점용
- 기존 영업장 인수
- 서류 준비 상태 확인

---

## 2. GMS 기반 Slot Filling

사용자 입력은 먼저 GMS가 내부적으로 분석합니다.

이 단계에서는 사용자에게 결과를 그대로 보여주지 않고, 다음 판단을 위해 필요한 구조화된 정보만 추출합니다.

주요 추출 항목은 아래와 같습니다.

| 분류 | 추출 예시 |
| --- | --- |
| 위치 | 구, 동, 도로명주소, 지번주소, 층, 호수 |
| 업종/목적 | 카페, 휴게음식점, 일반음식점, 제과점, 간판 설치 등 |
| 판매 품목 | 음료, 디저트, 음식, 주류 |
| 면적 | 15평, 50㎡ 등 |
| 시설 계획 | 간판, 외부 테이블, 조리시설, 좌석 |
| 권리 관계 | 임대차, 건물주/관리인 동의 |
| 서류 상태 | 임대차계약서, 위생교육 수료증, 건강진단결과서 등 |

중요한 점은 이 slot filling 결과는 사용자에게 노출되는 최종 화면이 아니라는 것입니다.

사용자에게는 "AI가 내부적으로 이해한 정보"가 아니라, 실제로 판단에 필요한 질문과 결과만 보여줍니다.

---

## 3. 부족 정보 질문

AI가 추출한 정보 중 판단에 꼭 필요한 값이 비어 있으면, UI에서 추가 질문을 합니다.

예시 질문:

- 술도 판매하나요?
- 매장에서 조리하나요?
- 간판을 설치할 예정인가요?
- 가게 앞 테이블을 둘 예정인가요?
- 외부 테이블은 보도/도로 위인가요, 사유지인가요?
- 임대차계약서나 사용 권한을 확인할 수 있나요?

이 단계의 목적은 사용자를 귀찮게 만드는 것이 아니라, 최종 서류와 가능 여부 판단에 필요한 핵심 정보만 보완하는 것입니다.

---

## 4. 주소 기반 API 조회

정확한 주소가 확보되면 공공 API를 호출합니다.

특히 건축물 용도는 사용자에게 직접 묻는 것이 아니라, 주소를 기반으로 건축물대장 API에서 확인하는 것이 원칙입니다.

API/DB 조회 대상:

- 주소 정규화
- 건축물대장 조회
- 주용도 확인
- 층별 용도 확인
- 면적 확인
- 위반건축물 여부 확인 가능 범위 검토
- 동일 장소 과거 영업 이력 조회 가능 범위 검토

예를 들어 카페 창업의 경우, 건축물대장상 용도가 휴게음식점 또는 근린생활시설 기준에 맞는지 확인해야 합니다.

---

## 5. 업종 가능 여부 판단

API 결과와 사용자 입력을 바탕으로 업종 가능 여부를 판단합니다.

판단 예시:

- 휴게음식점 가능성이 높음
- 일반음식점은 추가 조건 확인 필요
- 주류 판매를 하면 휴게음식점으로는 어려움
- 외부 테이블은 도로점용 또는 사유지 여부 확인 필요
- 간판은 옥외광고물 신고/허가 대상 여부 확인 필요

판단 결과는 단순히 "가능/불가능"만 보여주는 것이 아니라, 왜 그렇게 판단했는지 근거도 함께 보여줍니다.

---

## 6. 사용자 이해 확인

판단 전에 시스템은 사용자의 상황을 요약해서 보여줍니다.

예시:

```text
마포구 망원동에서 15평 규모의 카페를 창업하려고 하고,
음료와 디저트를 판매하며, 간판 설치를 계획 중인 것으로 이해했어요.
```

사용자는 이 요약이 맞으면 다음 단계로 진행하고, 틀리면 수정할 수 있습니다.

이 단계는 AI가 잘못 이해한 상태로 서류 안내까지 진행하는 것을 막기 위한 안전장치입니다.

---

## 7. 그래프/DB 기반 서류 리스트 생성

최종 서류 리스트는 GMS가 임의로 생성하지 않습니다.

서류 목록은 기존에 구축한 그래프와 SQLite DB를 기준으로 가져옵니다.

사용 데이터:

| 데이터 | 역할 |
| --- | --- |
| `final_graph` | 인허가 절차, 서류, 조건 관계 |
| `document_issue_guide.sqlite` | 서류 발급처, 제출처, 제출 시점, 선행서류 |
| 건축물대장 API 결과 | 업종 가능 여부와 조건 판단 |
| 사용자 추가 답변 | 간판, 외부공간, 주류, 조리 방식 등 조건 확정 |

즉, GMS는 서류를 새로 지어내는 역할이 아니라, 그래프/DB에서 나온 서류를 빠뜨리지 않고 정리하는 역할입니다.

---

## 8. 서류 순서 정리

그래프/DB에서 나온 서류들은 준비 순서가 중요합니다.

예를 들어 영업신고증 발급 전에는 아래 서류가 먼저 필요할 수 있습니다.

- 위생교육 수료증
- 건강진단결과서
- 임대차계약서 또는 사용 권한 증빙
- 신분증
- 소방완비증명서

그리고 사업자등록증은 일반적으로 영업신고증 발급 이후에 진행됩니다.

서비스는 아래 기준으로 서류를 정리합니다.

- 먼저 준비해야 하는 서류
- 동시에 준비 가능한 서류
- 영업신고 전에 필요한 서류
- 사업자등록 전에 필요한 서류
- 조건부로 필요한 서류
- 나중에 확인해도 되는 서류

---

## 9. 최종 결과 화면

사용자는 최종적으로 아래 정보를 확인합니다.

- 창업 또는 변경 가능성 판단
- 판단 근거
- 아직 부족한 정보
- 준비해야 할 서류 목록
- 서류별 발급처
- 서류별 제출처
- 제출 전 먼저 필요한 서류
- 준비 순서

현재 방향에서는 문의/부서 안내는 제거하고, 서류와 판단 흐름에 집중합니다.

---

## 현재 GMS 사용 지점

현재 GMS가 사용되는 부분은 아래와 같습니다.

| 단계 | GMS 사용 여부 | 설명 |
| --- | --- | --- |
| 자연어 slot filling | 사용 | 사용자 입력에서 주소, 업종, 조건, 서류 상태 추출 |
| 가능 여부/최종 판단 | 사용 | API/DB/그래프 결과를 바탕으로 설명 생성 |
| 서류 리스트 생성 | 직접 생성하지 않음 | 그래프/DB 기반으로 가져옴 |
| 서류 순서/설명 정리 | 보조적으로 사용 | 누락 없이 사용자 친화적으로 정리 |
| 문의/부서 안내 | 현재 제외 | 서비스 범위에서 일단 제거 |

현재 런타임 모델은 `gpt-5.5`이며, GMS를 통해 호출됩니다.

---

## 핵심 설계 원칙

허가온 V3의 핵심은 "AI가 모든 것을 마음대로 판단하는 서비스"가 아닙니다.

역할은 아래처럼 나뉩니다.

| 역할 | 담당 |
| --- | --- |
| 자연어 이해 | GMS |
| 주소/건축물 정보 확인 | 공공 API |
| 업종 가능 여부 기준 | 판단 로직 + API 결과 |
| 필요한 서류 관계 | 그래프/SQLite DB |
| 서류 순서와 안내 문장 | GMS 보조 |

정리하면 전체 흐름은 아래와 같습니다.

```text
사용자 자연어 입력
  ↓
GMS slot filling
  ↓
부족 정보 질문
  ↓
주소 기반 API 조회
  ↓
업종 가능 여부 판단
  ↓
사용자 이해 확인
  ↓
그래프/DB 기반 서류 확정
  ↓
GMS가 순서와 설명 정리
  ↓
최종 체크리스트 제공
```
